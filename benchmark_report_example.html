Glosario & Términos Técnicos

Término Significado
TPS Transacciones Por Segundo (Throughput). Métrica clave de rendimiento que mide cuántas operaciones de escritura se completan en un segundo.
MVCC Multi-Version Concurrency Control. Mecanismo que permite múltiples versiones de datos coexistir, permitiendo lectores ver versiones consistentes sin bloqueos.
n_dead_tup Número de tuplas (filas) muertas. En PostgreSQL, versiones antiguas de filas que han sido actualizadas pero aún ocupan espacio en disco esperando limpieza por VACUUM.
rel_size_bytes Tamaño de la relación (tabla) en bytes. Crece cuando se acumula bloat (espacio muerto) por filas muertas no limpiadas.
history_list_length Longitud de la cola de historial. En MySQL InnoDB, número de versiones antiguas almacenadas en el Undo Log esperando ser purgadas.
Write Amplification Amplificación de escritura. En PostgreSQL, cada UPDATE crea una nueva versión de la fila (append-only), causando que el almacenamiento físico crezca más que el cambio lógico.
Autovacuum Proceso automático de fondo en PostgreSQL que limpia filas muertas y recupera espacio. Si la tasa de UPDATEs excede la velocidad de limpieza, ocurre bloat.
Undo Log Registro de deshacer. En MySQL InnoDB, estructura que mantiene versiones antiguas de filas para propósitos de MVCC. Debe ser purgado periódicamente.
In-Place Update Actualización en lugar. En MySQL, la fila se modifica en su ubicación física original; versiones antiguas se almacenan en el Undo Log, no en la tabla.
Append-Only Solo añadir. En PostgreSQL, UPDATEs siempre crean nuevas versiones de la fila; la vieja versión se marca como muerta pero permanece en la tabla.
Page Contention Contención de páginas. En MySQL, múltiples UPDATEs a la misma fila compiten por acceso exclusivo a la página del buffer pool, limitando concurrencia.
Snapshot Isolation Aislamiento por fotografía. Cada transacción ve un estado consistente de datos en el momento en que comenzó, proporcionado por MVCC.

🔧 Configuración del Benchmark

Este benchmark fue diseñado para aislar y medir el impacto arquitectónico de las estrategias MVCC en ambos motores de bases de datos bajo una carga de escritura sostenida e intensa.

Parámetros de ejecución:

  • Número de filas insertadas inicialmente
  • Número de clientes concurrentes (workers) ejecutando UPDATEs en paralelo
  • Duración total de la prueba (segundos)
  • Tipo de operación: SELECT aleatorio de filas, seguido de UPDATE en contador
  • Modo de transacción: Autocommit (cada UPDATE es su propia transacción)

Métricas recolectadas cada 10 segundos:

  • PostgreSQL: n_tup_upd, n_dead_tup, pg_relation_size()
  • MySQL: SHOW ENGINE INNODB STATUS (extrae History list length)
  • Ambos: TPS por intervalo de 5 segundos durante la prueba

🔍 Analysis & Findings

Performance Summary

PostgreSQL achieved an average throughput of 5492 TPS with a peak of 6009 TPS, while MySQL (InnoDB) achieved 2290 TPS average with a peak of 2428 TPS.

💡 Performance Ratio
PostgreSQL delivered approximately 2.4x higher throughput than MySQL in this workload.

PostgreSQL: Write Amplification & Bloat

PostgreSQL's append-only MVCC design stores new row versions alongside old ones. Under sustained UPDATE load:

  • Dead Tuples Accumulation: The benchmark shows growth in n_dead_tup, indicating old row versions persist on disk after being superseded.
  • Table Bloat: As seen in the metrics, rel_size_bytes increases, meaning the physical table grows even though the logical row count remains constant.
  • Autovacuum Pressure: Postgres must continuously run background vacuum processes to reclaim space. If UPDATE rate exceeds vacuum speed, bloat compounds.
⚠️ Key Observation
PostgreSQL's design trades write amplification for snapshot isolation. Every UPDATE creates a new tuple version, requiring eventual cleanup. This is ideal for OLTP with many readers but can degrade under extreme UPDATE-heavy workloads.

MySQL (InnoDB): Undo Log Contention

MySQL modifies rows in-place and maintains version history in the Undo Log. Under sustained UPDATE load:

  • History List Length: Tracks pending versions in the Undo Log. High values indicate purge threads cannot keep up with transaction volume.
  • Page Contention: Since all versions of a row occupy the same physical page, concurrent UPDATEs contend on the same lock word, limiting scalability.
  • Undo Tablespace Growth: The ibdata1 or dedicated undo tablespaces expand as history accumulates, potentially consuming significant disk.
⚠️ Key Observation
MySQL's in-place update strategy is more cache-efficient for reads but suffers under high concurrency on the same rows due to lock contention on buffer pool pages.

MVCC Trade-offs Summary

Dimension PostgreSQL (Append-Only) MySQL (In-Place)
Write Strategy New versions appended; old versions marked dead Rows modified in place; versions stored in Undo Log
Disk Growth High (table bloat if vacuum lags) Moderate (undo tablespace grows)
Read Performance Excellent (no in-page fragmentation) Good but may scatter versions
Concurrent Updates High throughput (no page-level lock contention) Page-level contention limits scaling
Background Work Autovacuum (CPU-intensive cleanup) Purge threads (less disruptive)
Long Transactions Holds many dead rows in memory Holds undo log entries (can stall purge)

Recommendations

Choose PostgreSQL when:

  • High read concurrency is needed
  • Many concurrent UPDATE/INSERT operations from different clients
  • Long-running read-only transactions must see consistent snapshots
  • Disk space is plentiful and autovacuum can keep up

Choose MySQL (InnoDB) when:

  • UPDATE workload is moderate and read-heavy
  • Memory-to-disk ratio is constrained
  • Predictable, low-contention row access patterns
  • Undo log overhead can be managed via tuning
